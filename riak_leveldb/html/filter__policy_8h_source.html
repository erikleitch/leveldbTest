5 // A database can be configured with a custom FilterPolicy object.
6 // This object is responsible for creating a small filter from a set
7 // of keys. These filters are stored in leveldb and are consulted
8 // automatically by leveldb to decide whether or not to read some
9 // information from disk. In many cases, a filter can cut down the
10 // number of disk seeks form a handful to a single disk seek per
11 // DB::Get() call.
12 //
13 // Most people will want to use the builtin bloom filter support (see
14 // NewBloomFilterPolicy() below).

65 // Return a new filter policy that uses a bloom filter with approximately
66 // the specified number of bits per key. A good value for bits_per_key
67 // is 10, which yields a filter with ~ 1% false positive rate.
68 //
69 // Callers must delete the result after any database that is using the
70 // result has been closed.
71 //
72 // Note: if you are using a custom comparator that ignores some parts
73 // of the keys being compared, you must not use NewBloomFilterPolicy()
74 // and must provide your own FilterPolicy that also ignores the
75 // corresponding parts of the keys. For example, if the comparator
76 // ignores trailing spaces, it would be incorrect to use a
77 // FilterPolicy (like NewBloomFilterPolicy) that does not ignore
78 // trailing spaces in keys.
